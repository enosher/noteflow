## Known Issues

| ID | Found by case | Severity | Description | Decision & rationale |
| :- | :------------ | :------- | :---------- | :-------------------- |
| REC-01-MISMATCH | REC-01 | S2 | The recommendation algorithm (`lib/recommender.ts`: 40% topic weakness, 30% mistake recency, 20% recency boost, 10% difficulty match) picked a Polymorphism question over a Recursion one on the seeded demo account, even though Recursion is the weaker topic (40% vs 50%) and `scripts/seed-demo.ts` explicitly promises "Recommendations: should surface a Recursion question." Root cause: Recursion's "no reachable base case" question had its most recent (wrong) attempt seeded only 1 day ago, which zeroed out its recency-boost term (the window is 7 days), while Polymorphism's wrong attempt at 8 days ago maxed out both mistake-recency and recency-boost. The algorithm itself was working as coded - only the seed data's timestamp didn't line up with its own narrative. | **Fixed** (17 Jul 2026): changed that attempt's `daysAgo` from 1 to 8 in both `scripts/seed-demo.ts` and `lib/seed-data.ts`, so recency-boost fires alongside mistake-recency as originally intended. Verified via `scoreQuestion()` directly: Recursion's question now scores 0.8325 vs Polymorphism's 0.7925 (was 0.6325 before the fix). Needs a fresh `npx tsx scripts/seed-demo.ts` run against the live demo account to take effect there. |
| REV-04-STALE | REV-04 | S3 | After grading the last due card, the completion screen itself is correct ("all caught up", right session count), but the static subheading above it ("`N` questions due for review") and the nav bar's Review badge don't update until a full page reload — they keep showing the pre-completion count for the rest of that page view. | Cosmetic only; no user-facing harm since the actual queue and dashboard counts are correct once you navigate anywhere else. Leave as-is per severity policy (S3, no code changes after freeze) unless there's spare time. |
